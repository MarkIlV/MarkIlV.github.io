一、SSTI（Server-Side Template Injection）原理

要了解什么是SSTI，首先要说一下什么是模板引擎，下面是百度百科的解释：

模板引擎（这里特指用于Web开发的模板引擎）是为了使用户界面与业务数据（内容）分离而产生的，它可以生成特定格式的文档，用于网站的模板引擎就会生成一个标准的HTML文档。

也就是说，使用模板引擎可以将（网站）程序实现界面与数据分离，业务代码和逻辑代码分离，提升开发效率，同时也使得代码重构变得容易。

常见的一些模板引擎：

  • PHP

    • Smarty
    • Twig
    • Blade

  • Java

    • JSP
    • FreeMarker
    • Velocity

  • Python

    • Jinja2

    • dijango

    • tornado

网站应用程序通常会使用一些模板系统进行动态内容的嵌入，而当用户的输入以不安全的方法嵌入到模板中时，就有可能发生模板注入，比如一个使用Twig 模板来问候收件人的邮件服务，如果变量只是传入到模板之中，那么没啥问题：

而如果开发者信任了用户的输入而直接在模板内未经过滤的GET一个变量，那么就有可能引起模板注入：

可以看到，这里用户通过传入非法的custom_email变量控制了模板本身的内容，从而实现了代码的执行，而这里的代码实际上是在沙箱中执行的，但根据所使用的模板引擎，可以结合沙箱逃逸技术而实现任意代码执行。

2.识别阶段

检测到模板注入漏洞后，就要开始识别产生漏洞的模板引擎，当有报错信息时，我们有时候可能可以通过报错来确认其所使用的模板引擎，但当异常被捕捉时这种方法就没用了，以下是Burpsuite对不同模板接受的Payload的分类：

绿线表示成功返回，红线失败，同一个Payload在不同的模板引擎中可能返回不同的结果，例如 {{7*'7'}}在Twig中返回49，而在Jinja2中返回 7777777

3.利用阶段

当我们成功识别产生漏洞的模板引擎后就进入了漏洞利用阶段，在这个阶段，第一步要做的就是找到并阅读该模板引擎的文档，主要可以关注以下几个方面：

  • 该模板的基本语法
  • 安全注意事项——开发者可能会忽略该部分的内容，而我们在利用时也许会从中get到新姿势
  • 内置方法、函数、过滤器和变量
  • 插件/扩展——尤其要关注默认开启的

假设我们没有发现漏洞，那么下一步就是要去探索环境，找出当前环境可利用的函数或对象，许多模板引擎定义了一个self或namespace对象，或者可以列出所有属性和方法的函数

如果没有的话，那么我们需要爆破变量名

走到这一步时，我们应该可以去实时攻击了
